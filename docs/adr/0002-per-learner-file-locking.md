# Per-learner file locking, enabling multi-process workers

`state.py`, `surface_store.py`, `lesson_store.py`, and most of `workspace.py` do unlocked read-modify-write against a learner's files (`_load()` → mutate → `_save()`/`write_text()`). That was safe under the old single-seat model (`surface_store.py`'s docstring said so outright: "exactly one user is current per running process"), and stayed mostly safe under ADR 0001's per-request cookie identity, because FastAPI ran everything in one process's threadpool. It stops being safe once the server runs as multiple OS-process workers (`server.py`'s `serve()`, for `--workers N`): worker processes share no memory, so two tabs of the same learner — or two worker processes both handling that learner's requests — can race a read-modify-write cycle with no coordination at all, silently losing writes or corrupting a file.

We added `users.learner_lock()`: a single OS advisory lock (`fcntl.flock`) per learner, held for the whole duration of one turn (`harness.py`'s `loop()`) or one `/history` read. We chose one coarse per-learner-per-turn lock over fine-grained per-file locks in each module (state.py, surface_store.py, etc.) — every tool a turn runs writes this same learner's files, so a turn-scoped lock already serializes all of them without touching workspace.py's dozen call sites individually. The cost is that a second concurrent turn for the same learner queues behind the first rather than interleaving; we judged that acceptable since one learner sending two messages at once (two tabs, a slow connection retried) is rare, and serializing them is the *correct* behavior anyway — better than corrupting the transcript. `flock` releases automatically if the holding process dies, so a crashed worker can't leave a learner permanently locked out. Different learners never contend: each lock lives inside that learner's own home directory.

Also fixed as part of this: `/history` used to `load()` into a module-level `State()` shared by every request — a genuine cross-learner data leak under concurrent requests (FastAPI runs sync handlers in a threadpool even with one worker), independent of the workers question. It now uses a request-local `State()`.

We verified `learner_lock()` actually serializes across separate OS processes (not just threads) with a standalone multiprocessing test before enabling `workers>1` in `serve()` — this lock is a hard prerequisite for that flag, not an optional hardening.
